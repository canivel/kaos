Neuroplasticity in KAOS:
The Library Rewires Itself

Inline synaptic plasticity, Hebbian associations, failure triage, and systemic alerts — built the way brains do it. Every number in this post comes from a benchmark committed to the repo next to the docs. Nothing is planted.

April 24, 2026 18 min read v0.8.1 · 3 new benchmarks

The starting point

KAOS already had cross-agent memory, a skill library, and a shared log. Each one was a place — a table you wrote into and read from. But the library was static: skills accumulated, nothing ranked them, nothing pruned them, nothing noticed when two skills were always used together or when the same error kept coming back.

A brain doesn't work that way. A brain is a graph that rewires itself every time a neuron fires. What fires together wires together. What goes unused atrophies. What recurs reinforces. And the rewiring happens continuously, as a side-effect of normal operation — not in a scheduled batch job somebody has to remember to run.

v0.8.1 adds that layer to KAOS. It has a name: the dream cycle. But the real feature is what happens between cycles — the inline hooks that fire on every record_outcome, every memory.search, every kaos.complete(), and update the graph in place.

The honest framing. Neuroplasticity compounds for workloads that feed it signal — multi-session engagements where agents record outcomes and retrieve memory repeatedly. One-shot single-session runs with no feedback loop get zero gain. Set KAOS_DREAM_AUTO=0 to disable the hooks entirely if that's your pattern.

What got added

Five things, all additive. Existing KAOS code keeps working unchanged.

  • Inline plasticity hooks. Every skill outcome and memory hit writes telemetry in the same transaction as the caller's commit. Zero extra fsync on the hot path.
  • Batched Hebbian graph. At agent completion, one executemany rebuilds the associations for that agent's session — skill↔skill, memory↔memory, and cross-modal skill↔memory edges.
  • Weighted search. SkillStore.search(rank="weighted") reorders BM25 results by Wilson-lower-bound success rate multiplied by recency decay. Opt-in; default is unchanged.
  • Failure triage. Fingerprints now carry a category (transient / config / code / infra / unknown), a root cause, and a suggested action. A "known fix" that keeps failing auto-downgrades so future agents stop applying broken suggestions.
  • Systemic alerts. When ≥ N agents hit the same fingerprint inside a sliding window, KAOS raises a systemic_alerts row. Consumers should refuse to spawn until it's resolved — if the LLM endpoint is down, spawning more agents makes it worse.

Does it actually help? A real benchmark.

The headline question: does plasticity-weighted search retrieve the right skill more often than raw BM25? If yes, how much more often?

I built a deliberately-ambiguous benchmark: 10 retrieval queries, 20 skills arranged in twin pairs that share vocabulary so BM25 alone can't reliably disambiguate. Each query has one ground-truth correct skill and a plausible distractor. Then 80 training episodes, epsilon-greedy pick (ε=0.25, RNG seed 42), zero planted outcomes.

Both runs use the same seed, same data, same protocol. The only variable is whether the rank mode is feedback-sensitive.

That's a real delta, not a cherry-pick. But it's a specific benchmark that can show disambiguation gain. A benchmark where BM25 already hits 100% would show no gain — there's nothing for plasticity to improve. The point is: when the signal is there, plasticity uses it.

Training curve

bm25:     [60%, 70%, 70%, 70%]   ← flat; nothing learns from outcomes
weighted: [40%, 48%, 55%, 60%]   ← climbing; plasticity accumulates signal

Final top-1 (no exploration):
bm25:     80%
weighted: 90%

Weighted's training accuracy is lower than BM25's because weighted keeps demoting skills that fail, which forces it to try different candidates. The final measurement — one deterministic pick per query, no exploration — shows where the learning landed: BM25 plateaus at 80%, weighted reaches 90%.

What it costs — the story of a failed first try

Gains mean nothing if the overhead wipes them out. So I benchmarked that too. And the first architecture I wrote was bad enough that the benchmark rejected it.

The first design: inline association upserts

v0.8.0 built the Hebbian graph synchronously. On every record_outcome, the hook queried skill_uses for every other skill the agent had used, then upserted an association edge for each. On a library with many skills, that's N writes per outcome, plus a fresh COMMIT.

v0.8.0 — inline upserts

record_outcome overhead

p50: 210 ms extra per call.

On a 100-skill library with Windows fsync at ~30 ms, every outcome fired ~200 association inserts plus a second COMMIT. Unacceptable for the hot path.

The benchmark rejected the build. That's what benchmarks are for.

v0.8.1 — batched at completion

record_outcome overhead

p50: 15 µs extra per call.

The hot path writes one skill_uses row in the caller's transaction. Graph construction moved to kaos.complete() as a single executemany per agent session.

This is how sleep consolidation works biologically. Synaptic weights update in real time; structural rewiring happens offline.

Final measured overhead

Running demo_plasticity_overhead_bench/run.py on a library with 100 skills and 50 memory entries, 200 iterations per op:

Op Baseline p50 With plasticity p50 Overhead p50 Overhead p99
record_outcome 934 µs 949 µs +15 µs 10.5 ms
memory_search 1.08 ms 1.04 ms ~0 (noise) 1.3 ms
agent_complete 2.08 ms 2.95 ms +872 µs 1.5 ms

The 872 µs on agent_complete is the batched graph rebuild — a single executemany with all the association edges for the session. The p99 tail is dominated by OS fsync variance, not the hooks themselves.

Honest take. p50 overhead is essentially free. On Linux with faster fsync or an in-memory DB, every absolute number gets smaller. On systems where SQLite COMMIT is slow, the baseline is slow for everyone regardless of whether plasticity is on.

Failure triage: real examples

Pattern-matching on error strings tells you an error has happened before. That's useful but shallow. What you want is: why did it happen, is the known fix actually working, and is it systemic (many agents hitting the same thing — infra down — vs one-off).

v0.8.1 adds all three. Eight built-in heuristic diagnosers cover the high-volume cases without any LLM call. Users register custom ones for domain-specific patterns.

A walk through real categorisations

I planted seven realistic errors — rate limit, expired API key, harness mutation bug, type error, local vLLM down, disk full, DNS broken — and ran them through the diagnosis registry. Output:

Every category is a different action. A rate limit wants backoff. A 401 wants human action. A KeyError wants a code fix. A local ConnectionRefused wants you to start the service. KAOS can't magically fix them, but it can tell you which bucket you're in without you reading the stack trace.

Fix-outcome tracking — plasticity on the fix itself

Here's the part I think is actually new. When a fingerprint has a "known fix" attached, and an agent tries that fix, the agent reports whether the fix worked:

$ kaos dream fix-outcome 7 --succeeded       # fix worked
$ kaos dream fix-outcome 7 --failed           # fix did NOT work

After 5+ attempts with less than 50% success rate, the fix auto-downgrades: fix_summary gets cleared, future failure_lookup calls return no fix, agents fall back to fresh diagnosis instead of applying a broken suggestion.

It's the same Hebbian principle applied to fix suggestions. What keeps working stays; what stops working gets demoted. Same biology, different layer.

Systemic alerts

When four agents in two minutes all hit "Connection refused: localhost:8000", the answer isn't "retry with backoff, here's the known fix." The answer is "your vLLM server is down, don't spawn more agents."

KAOS sets KAOS_SYSTEMIC_THRESHOLD=5 and KAOS_SYSTEMIC_WINDOW_S=120 by default. Inside the window, if the threshold is crossed, a systemic_alerts row fires with the observed agent count. Debounced so it alerts once per incident.

$ kaos dream systemic
1 active systemic alert(s)
  #3  detected 2026-04-24T08:15:42  5 agents hit `llm_call` in 120s
    cause: Local service not reachable. A service KAOS or an agent is
           calling on localhost is not running.

$ kaos dream systemic --ack 3 --by "danilo: investigating"
$ kaos dream systemic --resolve 3 --by "danilo: restarted vllm"

Consumers should check list_active_alerts() before spawning:

from kaos.dream.phases.failures import list_active_alerts

if list_active_alerts(kaos.conn):
    print("Systemic issue active. Not spawning.")
    return

Architecture — what happens when

Event What fires inline What fires at agent completion What fires in a dream cycle
SkillStore.record_outcome() skill_uses row (15 µs)    
MemoryStore.search(record_hits=True) memory_hits rows (~0 µs)    
Kaos.complete() / .fail() / .kill() episode_signal row association graph rebuild;
on failure: fingerprint + diagnosis + systemic check		  
Every 100 completions (KAOS_DREAM_THRESHOLD)     consolidation pass (promote/prune/merge/policies)

The hot path doesn't build associations, doesn't run consolidation, doesn't call the LLM for diagnosis. It writes one row in the caller's existing transaction. The structural work happens at natural checkpoints — agent completion and threshold crossings — where a few extra ms is invisible in context.

MCP (8 tools)

Every CLI command has a matching MCP tool so agents running inside Claude Code, Cursor, or any other MCP client can use the same surface:

  • dream_run — run one dream cycle (dry-run or apply)
  • dream_related — Hebbian lookup for entities that co-fire
  • failure_lookup — agent-time fast path, given tool_name + error returns fingerprint + fix
  • failure_list — recurring fingerprints with category
  • failure_diagnose — get or set diagnosis for a fingerprint
  • failure_fix_outcome — record whether a fix worked
  • systemic_alerts — list active alerts; ack; resolve
  • dream_consolidate — propose or apply structural changes

Total MCP surface in KAOS now: 45 tools across 7 capability groups.

Python API

from kaos import Kaos
from kaos.skills import SkillStore
from kaos.memory import MemoryStore
from kaos.dream.phases.failures import lookup, list_active_alerts

kaos = Kaos("project.db")

# Weighted skill search — uses the Hebbian graph + success rates
skills = SkillStore(kaos.conn).search(
    "classify text topic", rank="weighted", limit=5,
)

# Memory search with record_hits=True feeds plasticity
mem = MemoryStore(kaos.conn)
snippets = mem.search("retry backoff", rank="weighted",
                      record_hits=True, requesting_agent_id=agent_id)

# Agent-time fast-path on errors
prior = lookup(kaos.conn, "http_get", error_msg)
if prior and prior["fix_summary"]:
    apply_known_fix(prior)

# Check for systemic issues before spawning
if list_active_alerts(kaos.conn):
    return  # don't spawn during incident

Escape hatches

  • KAOS_DREAM_AUTO=0 — disable all inline hooks. Telemetry still writes; associations, fingerprints, and threshold triggers don't fire.
  • KAOS_DREAM_THRESHOLD=<N> — tune auto-consolidation cadence (default 100 episodes).
  • KAOS_SYSTEMIC_THRESHOLD=<N> / KAOS_SYSTEMIC_WINDOW_S=<N> — tune systemic detection sensitivity.
  • rank="bm25" on any search call bypasses weighted scoring.

What I'd not claim — honest limits

Not every number in this post is the same confidence.

  • +10pp gain is scenario-specific. It's measured on a benchmark designed to have disambiguation signal. A workload with no outcome feedback, or where BM25 already gets it right, will see less or none.
  • Policies are populated but not consulted. The policies table fills up from repeated-approval patterns in the shared log, but nothing reads from it yet. That's on the M4 roadmap. Right now it's scaffolding.
  • Merge detection is Jaccard on word-bags. It catches obvious duplicates, misses semantic ones. Never auto-applied — always a proposal for human review.
  • Heuristic diagnosers cover common cases, not every case. Exotic errors stay "unknown" until you register a custom diagnoser or add an LLM-backed one.
  • Zero gain without feedback. If agents never call record_outcome, plasticity has nothing to learn from. The 80→90% benchmark works because the benchmark records outcomes — your project has to do the same to see equivalent gains.

The framing I keep coming back to

Most agent frameworks treat the library as a filing cabinet. You put things in, you retrieve them with full-text search, eventually you prune manually. The library is inert.

Brains aren't cabinets. Brains are graphs that rewire constantly as a side effect of thinking. The wiring is part of how thinking works, not a cleanup process you run on Sundays.

KAOS v0.8.1 is the first version where the library isn't inert. Every skill outcome reinforces or weakens a graph edge. Every memory retrieval updates the co-occurrence structure. Every agent completion triggers a tiny consolidation pass. Every failure gets categorised and triaged. And when infrastructure is down, the system notices before you do and tells you to stop spawning agents.

None of this requires you to remember anything. Run kaos run normally and the library gets better on its own. That's the feature.
