The Problem With Multi-Agent Systems

You're running multiple AI agents. Here's what goes wrong.

Agents step on each other

Two agents write to the same file. One overwrites the other's work. You don't find out until production.

You can't debug a rogue agent

It made 47 tool calls, modified 12 files, and now the codebase is broken. What did it actually do? In what order? Good luck with git log.

You can't roll back one agent

The refactor agent broke everything. You git reset --hard and lose the work of the 3 other agents that were fine.

State vanishes on crash

Agent crashes mid-task. Its progress, findings, intermediate files — all gone. Start over from scratch.

You can't inspect anything

How many tokens did each agent use? Which tool calls failed? What files did agent X touch? You're grep-ing through logs — if you have logs.

Your data leaves your machine

Every agent call hits a cloud API. Your proprietary code, internal docs, and trade secrets travel over the wire to someone else's servers.

How KAOS Solves This

 example.py 
from kaos import Kaos

db = Kaos("project.db")

# Each agent is isolated — they cannot see each other's files
agent_a = db.spawn("refactorer")
agent_b = db.spawn("test-writer")

db.write(agent_a, "/src/auth.py", b"# refactored auth module")
db.write(agent_b, "/src/auth.py", b"# test stubs for auth")
# Both wrote to "/src/auth.py" — no conflict. Each has their own copy.

# Checkpoint before risky operations
cp = db.checkpoint(agent_a, label="before-migration")
# ... agent does something dangerous ...
db.restore(agent_a, cp)  # roll back just this agent, others untouched

# What exactly did it do? Query the audit trail with SQL.
db.query("SELECT event_type, payload FROM events WHERE agent_id = ?", [agent_a])

Everything lives in one .db file. Copy it to back up. Send it to a teammate. Query with any SQLite client.

Key Capabilities

🔒

Enforced Agent Isolation

Every VFS operation is SQL-scoped with WHERE agent_id = ?. Physically impossible for one agent to access another's files. Optional FUSE tier adds OS-level mount + namespace isolation.

📜

Append-Only Audit Trail

14 event types cover every operation: file reads, writes, deletes, tool calls with timing and token counts, state changes, lifecycle events. Query with SQL.

⏪

Checkpoint / Restore / Diff

Snapshot files + state at any point. Restore to any checkpoint. Diff two checkpoints to see exactly what changed. Auto-checkpoints every N iterations.

🧠

Intelligent Model Routing

GEPA router classifies task complexity and picks the right model. Trivial task? 7B. Complex architecture? 70B. Works with any OpenAI-compatible endpoint.

📦

Content-Addressable Blobs

Files stored as SHA-256 blobs with zstd compression. Identical files across agents are deduplicated automatically. Reference counting with GC.

💾

Single-File Portability

The entire runtime is one .db file. Back it up with cp. Send it to a colleague. Query with DBeaver, DataGrip, or sqlite3. No cloud, no Docker.

🚀

Parallel Execution

Run up to 8 concurrent agents (configurable) with semaphore control. Thread-local DB connections. MVCC isolation via SQLite WAL mode.

🛠

Zero Bloat Dependencies

No openai, no litellm, no langchain. Just httpx, click, rich, textual, mcp, pyyaml, zstandard, ulid-py. 44 packages total.

Meta-Harness:
Your LLM Optimizes Its Own Wrapper

Changing the code around a fixed LLM produces a 6x performance gap. Meta-Harness automatically searches for the best harness — the prompt template, retrieval strategy, and memory management — by learning from full execution traces.

Based on arXiv:2603.28052 (Stanford / KRAFTON / MIT) — Original code

1

Evaluate Seed Harnesses

Start with 3 approaches (zero-shot, few-shot, retrieval). Each runs in its own isolated KAOS agent. Full execution traces — every LLM call, every output, every failure — are stored in the archive.

2

Proposer Reads Everything

An AI proposer agent reads ALL prior harness code, scores, AND raw execution traces. It identifies specific failure modes from the data — not guessing, not reading summaries, reading the actual traces. Median 82 files read per iteration.

3

Propose Targeted Fixes

The proposer submits new harness variants targeting the failures it found. Each is AST-validated, evaluated in an isolated KAOS agent, and stored with full traces. The paper shows purely additive changes outperform rewrites.

4

Pareto Frontier

After N iterations, you get the optimal tradeoffs: highest accuracy, lowest token cost, and everything in between. The entire search — every harness, trace, and proposer conversation — is one queryable .db file.

CheckpointRollback

Safe Refactoring with Rollback

Checkpoint before risky operations. If an agent breaks something, restore just that agent — other agents keep running untouched. Diff two checkpoints to see exactly what changed: files, state, tool calls.

Instant rollback • per-agent • other agents unaffected
cp = db.checkpoint(agent, label="pre-migration")
try:
    await ccr.run_agent(agent, "Migrate schema v3")
except Exception:
    db.restore(agent, cp)  # only this agent rolls back
